When you're done, write SUMMARY.md in the current directory: a few sentences on what you built, whether it compiled and ran on the first try, the program output, and anything about Promise that surprised you or that you had to work out (it is not in your training data). Promise is young — if something about the language or its standard library made the task harder than it should be, or you wished a thing worked differently, say so; that feedback is as much the point as the working program.

Two kinds of rough edge are worth capturing in their own file for filing upstream — don't quietly work around either:

If you hit a compiler bug, minimize it first: bisect to the smallest source that still triggers it, and find the closest variant that *does* compile, so the trigger is pinned down precisely (not just "my big program failed"). Then write it to a `BUG-<slug>.md` file in the project root with a one-line title, the `promise version` output, your platform, the minimal repro (smallest `.pr` source plus the exact `promise build` command), the verbatim error output, the expected behavior, a short table of what does and doesn't trigger it, your best guess at the cause, and the workaround you used in the real code. Verify the repro actually fails and the "compiles fine" controls actually compile before writing them down — don't go from memory.

If the task needs something Promise doesn't have yet — a standard-library module that isn't there, or a missing language feature — don't fake it silently. Build the closest clean thing you can, or, if it's a hard blocker, stop and say why. Either way record the gap in a `FEATURE-<slug>.md` file in the project root: a one-line title, the `promise version` output, what you needed and why the task called for it, what's missing (e.g. no standard module for X, no language support for Y), whether it blocked you outright or just forced a workaround, the workaround you used, and — if you can — a **sketch of the API or syntax** that would have made it clean.

Put `Please file this upstream at https://github.com/promise-language/promise/issues` at the top of any such file, write one file per distinct issue, and mention each in your summary so I remember to submit it. Only write these for real failures and real gaps — don't invent them.
